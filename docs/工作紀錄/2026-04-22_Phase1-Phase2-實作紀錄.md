### ③ `0806c8c` — fix: 重新綁定快捷鍵閃退（TSM 主執行緒斷言）

**變動**：`gui.py`（95+/5-）

**症狀**：設定 → 重新綁定 → App SIGTRAP 閃退。

**根因**：macOS 26.4+ 把 `TSMGetInputSourceProperty` 改成硬性要求必須在主 dispatch queue。pynput 在背景執行緒透過 ctypes 呼叫 TSM 解析鍵碼 → 斷言觸發 `brk 1`。

**崩潰堆疊**：
```
_dispatch_assert_queue_fail
  islGetInputSourceListWithAdditions
    TSMGetInputSourceProperty
      ffi_call_SYSV → _ctypes_callproc
        thread_run (Python 背景執行緒)
```

**修法**：`HotkeyBindDialog` 不再用 pynput Listener，改用 Tk 原生 `<KeyPress>` / `<KeyRelease>` binding。事件在主執行緒回調、完全不碰 TSM。

新增 `_keysym_to_name()` 把 Tk keysym（`Command_L`, `Alt_L`…）對應回 combo 命名。

主 `HotkeyManager`（全域 ⌘⌥R）沒動，避免碰同一顆雷。

**修正前 vs 修正後**：

```
  [修正前]                             [修正後]
  ─────────────                       ─────────────

  使用者點「重新綁定」                  使用者點「重新綁定」
      │                                   │
      ▼                                   ▼
  HotkeyBindDialog 建立                HotkeyBindDialog 建立
      │                                   │
      ▼                                   ▼
  _start_capture()                     _start_capture()
      │                                   │
      ▼                                   ▼
  threading.Thread(                    self.bind("<KeyPress>",
    target=capture_hotkey)               _on_tk_key_press)
      │                                   │
      ▼                                   ▼
  pynput Listener 在背景 thread        Tk 鍵盤事件 在主 thread
      │                                   │
      ▼  (macOS 26.4+)                    ▼
  某個鍵被按下                          某個鍵被按下
      │                                   │
      ▼                                   ▼
  pynput 用 ctypes 呼叫                Tk 事件直接回調
  TSMGetInputSourceProperty              │
      │                                   ▼
      ▼                              _on_tk_key_press
  ⚠ macOS dispatch_assert_queue        (主 thread，安全)
    強制必須在 main queue              ✓ 正常完成
      │
      ▼
  💥 SIGTRAP (brk 1) 閃退
```

### ④ `9e818fd` — fix: 重綁對話框無法重按 + 純修飾鍵靜默失敗

**變動**：`gui.py`（32+/27-）

**症狀** 1：按完第一組想換別的組合，再按沒反應。
**根因**：捕捉成功後 `_unbind_capture()` 直接解綁，後續按鍵全收不到。

**症狀** 2：「沒辦法儲存」—— 若使用者只按修飾鍵組合（如 ⌘⌥），落入 else 分支既不鎖定也不提示，Apply 永遠灰色。

**修法**：
- 成功後保持監聽、不 unbind
- `_reset_cycle()` 讓下一輪按鍵自動重置
- 純修飾鍵 release 顯式顯示「← 需要加字母/數字/空白」
- 移除 15 秒 timeout（捕捉成功後維持開啟）

**修正前 vs 修正後**（重按）：

```
  [修正前：按第一組後無法重按]          [修正後：可反覆嘗試]
  ─────────────────────────           ─────────────────────

  按 ⌘⌥R                               按 ⌘⌥R
  ⏎ → _captured="cmd+alt+r"           ⏎ → _captured="cmd+alt+r"
  ⏎ → _unbind_capture() ← 解綁         ⏎ → 保持 bind
                                        ⏎ → Apply 亮起
  想改意...
  按 ⌘⇧A                               想改意
  ⏎ → binding 已解除、事件收不到       按 ⌘⇧A
  ⏎ → UI 無反應、卡死                  ⏎ → _on_tk_key_press:
                                         if _captured and not current:

                                        ⏎ → _captured="cmd+shift+a"
                                        ⏎ → Apply 保持亮
```

### T10 小插曲（已釐清）

`prompt_reloader` 首次測試報錯。追查後確認是 Python 3.13 source loader 的**亞秒級 mtime 快取**：檔案在 1 秒內被改兩次會沿用 pycache 舊 bytecode。

**對實際使用無影響**（沒有人類會在 1 秒內改兩次 prompts.py），只是測試間隔要 > 1s。確認後重跑測試全過。

## 9 · 設計原則回顧（這次驗證有效的）

1. **永不阻塞主執行緒**：所有 HTTP I/O 設 timeout + 背景 thread。啟動 2s 非同步 health check 是關鍵決策。
2. **失敗降級鐵則**：OllamaClient 任何錯誤路徑都保證 `.text` 可用（填回原文），GUI 直接貼不必判斷。
3. **Lazy prompt lookup**：`prompts.ATTR` 動態查詢 + module-level string，是熱重載能運作的前提。
4. **狀態單一真相**：C2 的三段式 title、C1 的 `_last_raw` + `_last_llm_input` + `_last_polished`，都是「先設計狀態欄位、再寫 UI」的勝利。
