# 工作紀錄

每一次重大 session 結束留下的實作紀錄。給未來任何接手的開發者（包含未來的你自己、或下一個 Claude session）看的。

---

## 寫作原則

### 基本

- **繁體中文**
- **一個 session 一檔**，檔名格式：`YYYY-MM-DD_主題簡述.md`
- 盡量**當天寫當天的事**；補寫的話在內文標註

### 內容必備章節

1. **TL;DR** — 30 秒看懂這次做了什麼
2. **策略決策** — 為什麼選這條路（路徑分析、CEO review 決議…）
3. **Commit 時序** — 每筆 commit 逐筆摘要
4. **檔案盤點** — 新增／修改的檔案 + 依賴圖
5. **測試結果** — 自動化 + 使用者需實機驗證的清單分開
6. **剩餘 TODO** — 含優先度與工時估計
7. **下次 session 接手指南** — 直接能動工的切入點

## 🎯 流程圖慣例（鐵則）

**每一個 bug 修正、每一個架構重構，都必須附一張「修正前 vs 修正後」對照流程圖**。

### 為什麼

- 讀者（未來的你、或下一個接手的人）一眼看出「改了什麼」
- 用文字描述「把 X 改成 Y」很容易漏細節或講反方向
- 流程圖逼寫作者自己理清楚到底改了什麼

### 格式

用 **ASCII 流程圖**（而非 mermaid / 圖片），理由：
- 在任何 terminal / IDE / GitHub 都能正常顯示
- 不依賴渲染工具
- git diff 看得到差異

### 對照範本

```
[修正前]                         [修正後]
──────────────                   ──────────────
┌────────┐                       ┌────────┐
│ 入口   │                       │ 入口   │
└───┬────┘                       └───┬────┘
    │                                 │
    ▼                                 ▼
舊邏輯 ← ⚠ bug 點                 新邏輯 ✓
    │                                 │
    ▼                                 ▼
爆炸                              正確輸出
```

或縱向並排、或用表格對照關鍵步驟。只要**能讓讀者三秒內看出差別**都可以。

### 適用範圍

| 情境 | 要畫？ |
|---|---|
| 修一個 bug | ✅ 一定要 |
| 架構重構 | ✅ 一定要 |
| 新增功能 | ✅ 至少一張「接入點」流程圖 |
| 純 UI 調色 / 文案修改 | ❌ 免 |
| 單純改註解 / rename | ❌ 免 |

### 最小可接受範例

即使兩行也可以，只要清楚。例如：

```
[修正前] select_preset(text, _paste_target)   ← _paste_target 只在 auto_paste 開時才有值
[修正後] select_preset(text, _frontmost_app)  ← _frontmost_app 永遠抓，獨立於 auto_paste
```
